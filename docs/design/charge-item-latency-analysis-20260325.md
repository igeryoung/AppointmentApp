## Problem Statement
Creating and editing charge items feels slow in the current app. The user-visible delay happens after submit, not while typing in the dialog. The likely goal is to make charge-item mutations feel instant without giving up the current server-authoritative data model.

## What Makes This Cool
The best version of this flow feels local-first: tap save, the list updates immediately, totals change immediately, and sync happens quietly in the background. The user should not feel the cost of record resolution, network auth checks, or server-side flag maintenance.

## Constraints
- Charge items are now server-backed, not purely local.
- The server enforces device auth, book/record authorization, and version conflict checks.
- Event-level `has_charge_items` must stay correct.
- Existing offline fallback behavior matters; the current code already marks items dirty when server sync fails.
- This analysis is based on code-path inspection, not a measured on-device profile.

## What Already Exists
- The local-first pieces already existed in `ChargeItemOperationsMixin`: local save/delete, dirty flags, and event-flag maintenance.
- The controller already had best-effort sync helpers and offline state tracking; the fix reuses that shape instead of introducing a second mutation system.
- Existing tests already covered charge-item linkage and event-flag behavior, especially the controller unit coverage and `EVENT-INTEG-015/016`.

## NOT in Scope
- Full mutation queue or batching framework: stronger long-term architecture, but unnecessary for this latency fix.
- Record-resolution latency on first add for legacy non-UUID records: real issue, but separate from the common create/edit path for already-linked events.
- Broader auth/access query redesign on the server: the per-event flag loop was the hot inefficiency worth removing in this pass; the rest of the route can be revisited later if profiling still points there.

## Premises
1. The biggest latency is synchronous waiting in the UI path, not dialog rendering.
2. The current save path does more work than necessary on both client and server for a single charge-item mutation.
3. Optimistic UI alone will improve perceived latency a lot, but it will not fix backend inefficiency that grows with larger records.

## Approaches Considered
### Approach A: Optimistic Local-First Save
Summary: Keep the current server API, but stop waiting for it before updating UI. Save to local DB, update controller state immediately, fire sync in background, and reconcile only on failure/conflict.

Effort: M
Risk: Med

Pros:
- Biggest perceived UX win for the least code churn.
- Reuses the existing dirty-flag and best-effort sync design.
- Keeps offline behavior aligned with the current architecture.

Cons:
- Does not reduce server cost; it only hides it from the user.
- Conflict handling needs a clear UX path because the user may briefly see optimistic data.
- Still does duplicate local persistence unless the local reconciliation path is cleaned up too.

Reuses:
- `saveChargeItem` + dirty local write in the controller.
- Existing `ApiConflictException` path.
- Existing local DB schema and offline flags.

### Approach B: Thin Client + Cheaper Server Mutation
Summary: Keep the current user-visible behavior, but reduce mutation cost on the server and trim redundant client work. The server should update charge flags in one set-based operation, and the client should avoid double-applying the same mutation.

Effort: M
Risk: Low

Pros:
- Improves true end-to-end latency, not just perceived latency.
- Lowers cost for records with many events or many charged visits.
- Reduces duplicate DB work on both client and server.

Cons:
- The user still waits on the round-trip unless paired with optimistic UI.
- Requires touching both app and server code paths.
- Gains may be modest on very small records with a fast LAN.

Reuses:
- Current route structure and API contract.
- Current record/device authorization model.
- Existing event flag semantics.

### Approach C: Mutation Queue With Background Flush
Summary: Formalize charge-item writes as queued local mutations. The UI always commits locally first, then a background worker batches or serializes sync to the server.

Effort: L
Risk: Med

Pros:
- Best long-term architecture for responsiveness and offline reliability.
- Creates a consistent pattern reusable for other server-backed artifacts.
- Opens the door to batching server writes and retry logic.

Cons:
- Higher coordination cost and more moving parts.
- Requires careful conflict resolution and queue visibility.
- Bigger scope than needed if charge-item latency is the only pain point today.

Reuses:
- Current dirty-item model.
- Existing local charge-item storage.
- Existing server endpoints as the initial flush target.

## Recommended Approach
Choose Approach A plus the server cleanups from Approach B.

Reason:
- The user’s complaint is latency on create/edit, which is primarily a perception problem caused by awaiting sync in the foreground.
- The code already has the pieces for a local-first model, but it currently blocks on the server before the UI settles.
- Server inefficiency is real enough that it should be cleaned up in the same pass, but it should not block the first UX win.

Recommended implementation order:
1. Make create/edit/delete/toggle update local state immediately after `prdDb.saveChargeItem()` or `prdDb.deleteChargeItem()`.
2. Move server sync off the critical path and reconcile in the background.
3. Remove redundant local re-read when the controller already knows the new item state.
4. Simplify the server-side `has_charge_items` refresh into a set-based update instead of reset-all plus per-event loop.

## Root Cause Analysis
### 1. The UI waits for the full sync path before it refreshes
On add/edit/toggle/delete, the controller awaits `_saveChargeItemWithBestEffortSync()` and only then calls `loadChargeItems()`. That means the visible update waits for local save, credential lookup, HTTP request, server-side DB work, local reconciliation from the server response, and then a full local reload.

Evidence:
- `addChargeItem()` awaits sync, then `loadChargeItems()` in [event_detail_controller.dart](/Users/yangping/Studio/side-project/scheduleNote/lib/screens/event_detail/event_detail_controller.dart#L1276).
- `editChargeItem()` does the same in [event_detail_controller.dart](/Users/yangping/Studio/side-project/scheduleNote/lib/screens/event_detail/event_detail_controller.dart#L1323).
- `_saveChargeItemWithBestEffortSync()` performs local save first, then awaits server sync in [event_detail_controller.dart](/Users/yangping/Studio/side-project/scheduleNote/lib/screens/event_detail/event_detail_controller.dart#L1507).

### 2. Every mutation is effectively written locally twice
The client first writes locally with `prdDb.saveChargeItem()`, then sends the item to the server, then writes locally again with `prdDb.applyServerChargeItemChange(serverItem)`, then reloads all items for the record.

Evidence:
- First local write in [charge_item_operations_mixin.dart](/Users/yangping/Studio/side-project/scheduleNote/lib/services/database/mixins/charge_item_operations_mixin.dart#L82).
- Second local write in [event_detail_controller.dart](/Users/yangping/Studio/side-project/scheduleNote/lib/screens/event_detail/event_detail_controller.dart#L1498) and [charge_item_operations_mixin.dart](/Users/yangping/Studio/side-project/scheduleNote/lib/services/database/mixins/charge_item_operations_mixin.dart#L293).
- Full reload in [event_detail_controller.dart](/Users/yangping/Studio/side-project/scheduleNote/lib/screens/event_detail/event_detail_controller.dart#L1158).

### 3. The server mutation path is chatty and scales with record size
One save request performs multiple DB reads before the mutation, then recomputes event charge flags by clearing all events for the record and re-updating linked events one by one.

Evidence:
- Device auth read in [charge_item_routes.dart](/Users/yangping/Studio/side-project/scheduleNote/server/lib/routes/charge_item_routes.dart#L51).
- Device role read in [charge_item_routes.dart](/Users/yangping/Studio/side-project/scheduleNote/server/lib/routes/charge_item_routes.dart#L78).
- Record/book access reads in [charge_item_routes.dart](/Users/yangping/Studio/side-project/scheduleNote/server/lib/routes/charge_item_routes.dart#L98) and [charge_item_routes.dart](/Users/yangping/Studio/side-project/scheduleNote/server/lib/routes/charge_item_routes.dart#L122).
- Persisted event validation in [charge_item_routes.dart](/Users/yangping/Studio/side-project/scheduleNote/server/lib/routes/charge_item_routes.dart#L172).
- Insert/update branch in [charge_item_routes.dart](/Users/yangping/Studio/side-project/scheduleNote/server/lib/routes/charge_item_routes.dart#L355).
- Reset-all then looped flag refresh in [charge_item_routes.dart](/Users/yangping/Studio/side-project/scheduleNote/server/lib/routes/charge_item_routes.dart#L207).

This means the route cost grows with:
- the number of accessible books checked for the device,
- the number of events for the record,
- the number of distinct event IDs linked to charge items.

### 4. First-time add can pay an extra network round-trip before save
If the active record UUID is not already canonical, `ensureChargeItemsReady()` or `addChargeItem()` may call `getOrCreateRecord()` before the charge-item save even begins.

Evidence:

## Open Questions
- Is the reported latency mostly on add, mostly on edit, or equally visible on both?
- Does the slowdown become noticeably worse on records with long event history?
- Is the app usually talking to a local LAN server or a remote server over the internet?

## Success Criteria
- Create/edit/toggle/delete of a charge item updates the visible list in under 100 ms on a normal device.
- The UI does not wait for server round-trip to show the new value.
- Sync failures preserve local changes and surface clear offline/conflict state.
- Server route time no longer grows linearly with the number of linked charged events.

## Distribution Plan
Existing app deployment path covers this. No new artifact is needed.

## Next Steps
1. Instrument timings around local save, server save, local reconciliation, and `loadChargeItems()` to confirm the split on a real device.
2. Implement optimistic local state update in the charge-item controller methods.
3. Remove redundant list reload when the updated item is already known locally.
4. Refactor server flag refresh into one set-based SQL operation.
5. Re-measure on a record with many events to confirm the backend path no longer scales poorly.
